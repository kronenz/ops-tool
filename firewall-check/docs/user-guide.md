## 1. 개요

### 1.1 도구 설명

Firewall Connectivity Check Tool은 Kubernetes 클러스터 환경에서 방화벽 연결 상태를 진단하는 도구입니다. 클러스터 노드에서 대상 서버로의 네트워크 연결성을 체계적으로 검증하여 방화벽 정책 적용 여부를 확인합니다.

### 1.2 3단계 진단 프로세스

본 도구는 연결 문제의 원인을 정확히 파악하기 위해 3단계 계층적 진단을 수행합니다:

| 단계 | 계층 | 검증 내용 | 사용 도구 |
|------|------|-----------|-----------|
| 1단계 | Network | 라우팅 테이블 확인 | `ip route get` |
| 2단계 | Host | 호스트 도달 가능 여부 (ICMP) | `ping` |
| 3단계 | Port | 포트 오픈 여부 | `nc` (netcat) |

각 단계는 순차적으로 실행되며, 이전 단계가 실패하면 다음 단계는 생략됩니다. 이를 통해 문제가 발생한 정확한 계층을 식별할 수 있습니다.

### 1.3 주요 기능

- **병렬 실행**: 다중 노드에서 동시에 테스트를 실행하여 대규모 클러스터 환경에서도 빠른 검증 가능
- **다중 노드 지원**: 노드 목록 파일을 통해 여러 노드에서 일괄 테스트 수행
- **CIDR 기반 매칭**: 소스 IP를 CIDR 표기법으로 지정하여 서브넷 단위 관리 지원
- **다중 타겟/포트**: 하나의 규칙에 여러 타겟 IP와 포트를 콤마로 구분하여 지정 가능
- **실시간 진행률 표시**: 다중 노드 실행 시 각 노드의 진행 상황을 실시간으로 모니터링
- **Dry-run 모드**: 실제 네트워크 테스트 없이 시뮬레이션 실행
- **상세 결과 리포트**: CSV 형식의 상세 결과 및 요약 파일 생성

## 2. 설치 및 요구사항

### 2.1 필요 도구

본 도구는 다음 시스템 명령어를 필요로 합니다:

| 도구 | 용도 | 확인 명령어 |
|------|------|-------------|
| bash | 스크립트 실행 (v4.0 이상) | `bash --version` |
| ssh | 원격 노드 접속 | `ssh -V` |
| nc (netcat) | TCP/UDP 포트 테스트 | `nc -h` |
| ping | ICMP 연결 테스트 | `ping -V` |
| scp | 파일 전송 (다중 노드 모드) | `scp` |

### 2.2 SSH 키 설정 방법

다중 노드 모드를 사용하려면 컨트롤플레인 노드에서 대상 노드로 비밀번호 없이 SSH 접속이 가능해야 합니다.

**1단계: SSH 키 생성 (컨트롤플레인 노드에서 실행)**

```bash
ssh-keygen -t rsa -b 4096 -N "" -f ~/.ssh/id_rsa
```

**2단계: 공개 키 배포 (각 대상 노드로)**

```bash
ssh-copy-id user@target-node-ip
```

또는 수동으로 복사:

```bash
cat ~/.ssh/id_rsa.pub | ssh user@target-node-ip "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"
```

**3단계: 접속 확인**

```bash
ssh -o BatchMode=yes target-node-ip "echo OK"
```

## 3. 사용법

### 3.1 기본 실행 명령어

```bash
./bin/check_firewall.sh -i <csv_file> [옵션]
```

### 3.2 옵션 설명

| 옵션 | 필수 | 인자 | 설명 |
|------|------|------|------|
| `-i` | O | 파일경로 | 방화벽 규칙 CSV 파일 (필수) |
| `-N` | X | 파일경로 | 노드 목록 파일 (다중 노드 병렬 실행) |
| `-I` | X | IP주소 | 노드 IP 강제 지정 (자동 감지 대신 사용) |
| `-t` | X | 초 | 타임아웃 시간 (기본값: 2초) |
| `-o` | X | 디렉토리 | 결과 출력 디렉토리 (기본값: ./reports) |
| `-n` | X | - | Dry-run 모드 (실제 테스트 없이 시뮬레이션) |
| `-q` | X | - | Quiet 모드 (진행 출력 생략, 내부 사용) |

### 3.3 단일 노드 모드

현재 실행 중인 노드에서 테스트를 수행합니다. 노드의 IP는 자동으로 감지됩니다.

```bash
./bin/check_firewall.sh -i data/firewall_rules.csv
```

특정 IP를 소스로 지정하려면 `-I` 옵션을 사용합니다:

```bash
./bin/check_firewall.sh -i data/firewall_rules.csv -I 10.40.28.101
```

### 3.4 다중 노드 모드 (병렬 실행)

노드 목록 파일을 지정하여 여러 노드에서 동시에 테스트를 실행합니다:

```bash
./bin/check_firewall.sh -i data/firewall_rules.csv -N data/nodes.txt
```

이 모드에서는 다음 과정이 자동으로 수행됩니다:

1. 각 노드로 SSH 접속 확인
2. 스크립트와 CSV 파일을 원격 노드로 배포
3. 모든 노드에서 병렬로 테스트 실행
4. 결과를 컨트롤플레인 노드로 수집
5. 전체 결과 집계 및 리포트 생성

## 4. 입력 파일 형식

### 4.1 CSV 파일 형식

방화벽 규칙 CSV 파일은 파이프(`|`) 문자를 구분자로 사용합니다.

**필수 컬럼 (처음 5개):**

| 컬럼 | 설명 | 예시 |
|------|------|------|
| SERVICE | 서비스명 (식별용) | ETL, API, DNS |
| SOURCE | 소스 IP 또는 CIDR | 10.40.28.0/24, 192.168.1.100 |
| TARGET | 타겟 IP (콤마로 다중 지정 가능) | 10.25.200.87, 10.25.200.88 |
| PORT | 포트 번호 (콤마로 다중 지정 가능, ICMP는 비움) | 5432, 5433 |
| PROTOCOL | 프로토콜 (TCP, UDP, ICMP) | TCP |

**추가 컬럼 (선택사항):**

5번째 컬럼 이후는 무시되며, 요청일자, 요청자 등 관리 정보를 기록할 수 있습니다.

**샘플 CSV 파일:**

```
SERVICE|SOURCE|TARGET|PORT|PROTOCOL|요청일자|요청자|처리일자|처리자
ETL|10.40.28.0/24|10.25.200.87, 10.25.200.88|5432, 5433|TCP|26/01/16|변상현 책임|26/01/19|김현태 책임
API|10.40.28.0/24|10.25.200.90|443, 8080|TCP|26/01/17|이영희 책임|26/01/20|김현태 책임
DNS|10.40.28.0/24|10.25.200.91, 10.25.200.92|53|UDP|26/01/18|박철수 책임|26/01/21|김현태 책임
Monitoring|10.40.28.0/24|10.25.200.93|9090, 9091, 9100|TCP|26/01/19|김민수 책임|26/01/22|김현태 책임
Network|10.40.28.0/24|10.25.200.94, 10.25.200.95||ICMP|26/01/20|최지영 책임|26/01/23|김현태 책임
Batch|10.40.28.0/24|10.25.200.96|22|TCP|26/01/21|홍길동 책임|26/01/24|김현태 책임
```

### 4.2 각 필드 상세 설명

**SERVICE**
- 방화벽 규칙을 식별하는 서비스명
- 결과 리포트에서 규칙을 구분하는 데 사용
- 한글/영문 모두 사용 가능

**SOURCE**
- 테스트를 수행할 소스 IP 또는 CIDR 범위
- 노드 IP가 이 범위에 포함될 때만 해당 규칙이 테스트됨
- 예: `10.40.28.0/24` - 10.40.28.1 ~ 10.40.28.254 범위의 모든 노드에서 테스트

**TARGET**
- 연결 대상 IP 주소
- 여러 IP를 콤마(`,`)로 구분하여 지정 가능
- 각 IP에 대해 개별 테스트 수행

**PORT**
- 대상 포트 번호
- 여러 포트를 콤마(`,`)로 구분하여 지정 가능
- ICMP 프로토콜의 경우 비워둠
- 각 포트에 대해 개별 테스트 수행

**PROTOCOL**
- TCP: TCP 포트 연결 테스트
- UDP: UDP 포트 연결 테스트
- ICMP: Ping 테스트 (포트 미지정)
- 비어있으면 ICMP로 처리

### 4.3 노드 목록 파일 형식

노드 목록 파일은 한 줄에 하나의 IP 주소를 기록합니다.

**샘플 노드 목록 파일:**

```
10.40.28.101
10.40.28.102
10.40.28.103
```

**규칙:**
- 한 줄에 하나의 IP 주소
- `#`으로 시작하는 줄은 주석으로 처리
- 빈 줄은 무시됨

## 5. 실행 예시

### 5.1 기본 실행

현재 노드에서 방화벽 테스트를 실행합니다:

```bash
./bin/check_firewall.sh -i data/sample_firewall.csv
```

### 5.2 다중 노드 실행

여러 노드에서 동시에 테스트를 실행합니다:

```bash
./bin/check_firewall.sh -i data/sample_firewall.csv -N data/sample_nodes.txt
```

### 5.3 Dry-run 모드

실제 네트워크 테스트 없이 시뮬레이션을 실행합니다. CSV 파일의 형식 검증 및 예상 테스트 개수를 확인할 때 유용합니다:

```bash
./bin/check_firewall.sh -i data/sample_firewall.csv -n
```

다중 노드 dry-run:

```bash
./bin/check_firewall.sh -i data/sample_firewall.csv -N data/sample_nodes.txt -n
```

### 5.4 타임아웃 설정

네트워크 지연이 있는 환경에서 타임아웃을 늘립니다:

```bash
./bin/check_firewall.sh -i data/sample_firewall.csv -t 5
```

### 5.5 출력 디렉토리 지정

결과 파일을 저장할 디렉토리를 지정합니다:

```bash
./bin/check_firewall.sh -i data/sample_firewall.csv -o /tmp/firewall-results
```

### 5.6 전체 옵션 조합 예시

```bash
./bin/check_firewall.sh \
  -i data/firewall_rules.csv \
  -N data/cluster_nodes.txt \
  -t 3 \
  -o /var/log/firewall-checks
```

## 6. 출력 결과

### 6.1 콘솔 출력 설명

실행 시 5단계 프로세스가 화면에 출력됩니다:

```
┏━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃ 방화벽 3단계 진단 v3.2                                      ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛
  i  입력: data/sample_firewall.csv
  i  진단: [1]Network -> [2]Host -> [3]Port
```

### 6.2 결과 아이콘 의미

| 아이콘 | 상태 | 설명 |
|--------|------|------|
| `(초록)` | PASS | 테스트 성공 |
| `(빨강)` | FAIL | 테스트 실패 |
| `(회색)` | SKIP | 이전 단계 실패로 생략됨 |
| `(노랑)` | RUN | 테스트 실행 중 |
| `(회색 빈원)` | WAIT | 대기 중 |

### 6.3 결과 파일 설명

**report_[노드IP]_[타임스탬프].csv**

개별 테스트 상세 결과를 담은 CSV 파일입니다.

컬럼 구성:
| 컬럼 | 설명 |
|------|------|
| TIMESTAMP | 테스트 실행 시간 |
| SERVICE | 서비스명 |
| SOURCE | 소스 CIDR |
| NODE | 테스트 수행 노드 IP |
| TARGET | 대상 IP |
| PORT | 대상 포트 |
| PROTOCOL | 프로토콜 |
| RESULT | 최종 결과 (PASS/FAIL) |
| FAIL_AT | 실패한 단계 (Network/Host/Port) |
| NETWORK | Network 단계 결과 |
| HOST | Host 단계 결과 |
| PORT | Port 단계 결과 |

**summary_[타임스탬프].txt**

다중 노드 실행 시 전체 집계 결과를 담은 요약 파일입니다.

```
================================================================================
 방화벽 3단계 진단 - 전체 집계
================================================================================
 실행시각: 2026-01-23 15:30:45
 컨트롤플레인: 10.40.28.100
 대상노드: 3개 (성공: 3, 실패: 0)
--------------------------------------------------------------------------------
 총 테스트: 45 | PASS: 42 | FAIL: 3 | 성공률: 93%
================================================================================
```

## 7. 실행 프로세스

다중 노드 모드에서의 5단계 실행 프로세스를 상세히 설명합니다.

### 7.1 1단계: 노드 목록

노드 목록 파일을 읽어 대상 노드를 확인합니다.

```
──────────────────────────────────────────────────────────────
  1단계: 노드 목록
──────────────────────────────────────────────────────────────
  i  컨트롤플레인: 10.40.28.100
  i  대상 노드: 3개
    (초록) 10.40.28.100 (local)
    (파랑) 10.40.28.101 (remote)
    (파랑) 10.40.28.102 (remote)
```

- local: 현재 스크립트가 실행 중인 노드
- remote: SSH로 접속할 원격 노드

### 7.2 2단계: SSH 접근 확인

각 원격 노드로의 SSH 접속 가능 여부를 확인합니다.

```
──────────────────────────────────────────────────────────────
  2단계: SSH 접근 확인
──────────────────────────────────────────────────────────────
    10.40.28.100     (초록) local
    10.40.28.101     (초록) OK
    10.40.28.102     (초록) OK
```

SSH 접속 실패 노드는 테스트에서 제외되며, 결과 집계에서 0건으로 표시됩니다.

### 7.3 3단계: 파일 배포

스크립트와 CSV 파일을 원격 노드로 복사합니다.

```
──────────────────────────────────────────────────────────────
  3단계: 파일 배포
──────────────────────────────────────────────────────────────
    10.40.28.100     (초록) skip (local/dry)
    10.40.28.101     (초록) OK
    10.40.28.102     (초록) OK
```

- 로컬 노드는 배포 생략
- 배포 실패 시 해당 노드는 테스트에서 제외

### 7.4 4단계: 병렬 테스트 실행

모든 노드에서 동시에 테스트를 실행하며, 실시간으로 진행률을 표시합니다.

```
──────────────────────────────────────────────────────────────
  4단계: 병렬 테스트 실행
──────────────────────────────────────────────────────────────
  i  모든 노드에서 동시 실행 중...

  (노랑) 10.40.28.100 (45%)  (노랑) 10.40.28.101 (30%)  (초록) 10.40.28.102 (15/0)
```

진행률 표시:
- `(노랑) 노드IP (퍼센트%)`: 테스트 진행 중
- `(초록) 노드IP (PASS/FAIL)`: 테스트 완료

### 7.5 5단계: 결과 집계

모든 노드의 결과를 수집하여 표로 표시합니다.

```
──────────────────────────────────────────────────────────────
  5단계: 결과 집계
──────────────────────────────────────────────────────────────

+-----------------+----------+----------+----------+----------+
| NODE            | TESTS    | PASS     | FAIL     | RATE     |
+-----------------+----------+----------+----------+----------+
| (초록) 10.40.28.100  |       15 |       14 |        1 |     93%  |
```

## 8. 문제 해결

### 8.1 "매칭되는 SOURCE 없음" 오류

**증상:**

```
  경고  매칭되는 SOURCE 없음
  - 10.40.28.0/24
  - 192.168.1.0/24
```

**원인:**

현재 노드의 IP 주소가 CSV 파일에 정의된 어떤 SOURCE CIDR에도 포함되지 않습니다.

**해결 방법:**

1. 현재 노드의 IP 확인:
   ```bash
   hostname -I | awk '{print $1}'
   ```

2. CSV 파일의 SOURCE 컬럼에 현재 노드 IP가 포함되는 CIDR 범위가 있는지 확인

3. 필요시 `-I` 옵션으로 IP를 강제 지정:
   ```bash
   ./bin/check_firewall.sh -i data/rules.csv -I 10.40.28.101
   ```

### 8.2 SSH 연결 실패

**증상:**

```
    10.40.28.102     (빨강) FAIL
```

**원인:**

- SSH 키가 설정되지 않음
- 대상 노드의 SSH 서비스가 중지됨
- 네트워크 연결 문제
- 방화벽에서 SSH 포트(22) 차단

**해결 방법:**

1. SSH 키 설정 확인:
   ```bash
   ssh -o BatchMode=yes 10.40.28.102 "echo OK"
   ```

2. 수동 SSH 접속 테스트:
   ```bash
   ssh 10.40.28.102
   ```

3. 대상 노드의 SSH 서비스 확인:
   ```bash
   systemctl status sshd
   ```

4. 연결 디버그:
   ```bash
   ssh -vvv 10.40.28.102
   ```

### 8.3 타임아웃 문제

**증상:**

많은 테스트가 FAIL로 표시되거나 실행 시간이 과도하게 길어짐

**원인:**

- 네트워크 지연이 기본 타임아웃(2초)보다 큼
- 대상 서버 응답 지연
- 네트워크 혼잡

**해결 방법:**

1. 타임아웃 값 증가:
   ```bash
   ./bin/check_firewall.sh -i data/rules.csv -t 5
   ```

2. 특정 대상에 대해 수동 테스트:
   ```bash
   ping -c 3 -W 5 10.25.200.87
   nc -z -w 5 10.25.200.87 5432
   ```

### 8.4 CSV 파일 형식 오류

**증상:**

```
Error: Invalid CSV header
```

**원인:**

CSV 파일의 헤더가 올바른 형식이 아닙니다.

**해결 방법:**

1. 헤더가 `SERVICE|SOURCE|TARGET`으로 시작하는지 확인

2. 구분자가 파이프(`|`)인지 확인 (콤마나 탭이 아님)

3. 파일 인코딩 확인 (UTF-8 권장):
   ```bash
   file data/rules.csv
   ```

### 8.5 결과 파일이 생성되지 않음

**증상:**

테스트 실행 후 reports 디렉토리에 파일이 없음

**원인:**

- 출력 디렉토리 쓰기 권한 없음
- 디스크 공간 부족

**해결 방법:**

1. 출력 디렉토리 권한 확인:
   ```bash
   ls -la reports/
   ```

2. 디스크 공간 확인:
   ```bash
   df -h .
   ```

3. 다른 출력 디렉토리 지정:
   ```bash
   ./bin/check_firewall.sh -i data/rules.csv -o /tmp/firewall-results
   ```
